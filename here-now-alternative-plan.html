Proposal · analysis first · no implementation

Internal temporary hosting for agents

A deliberately small, single-container service that accepts HTML, Markdown, or a ZIP of prebuilt static files, returns a private-company URL immediately, and deletes the deployment after 30 days.

Executive recommendation

Build the static-hosting core, not the hosting platform

here.now has accounts, object storage, versions, access control, custom domains, analytics, and drives. None of that is needed for the first internal version.

Build

One immutable deployment endpoint

Each publish creates a new random ID and URL. There is no update, delete, list, account, or ownership workflow in the MVP. Expiry is always 30 days.

Accept

HTML, Markdown, and ZIP

Single-file input is convenient for agents. ZIP is the canonical format for a complete Vite, React, Vue, Svelte, or other static build output.

Run

One Go binary and one data volume

The container receives uploads, renders Markdown, serves static files, stores metadata, and runs the expiry sweep. An existing company ingress handles TLS.

Important boundary: accept the result of npm run build—for example dist/ or build/. Do not accept a Node.js source project and run npm install or build commands inside the hosting service. Executing uploaded projects turns a simple static host into a remote-code-execution platform.

MVP contract

What the service will and will not host

The narrow scope is intentional. It makes the service predictable for agents and safe enough for internal deployment.

Included

Supported input

  • HTML: a single document becomes index.html.
  • Markdown: rendered server-side into a standard, self-contained HTML shell.
  • ZIP: a full prebuilt static site with HTML, CSS, JavaScript, images, fonts, JSON, WASM, media, and source maps.
  • SPA fallback: optional for client-side routers.
  • Immutable URL: every POST creates a new deployment.
Excluded from MVP

Unsupported workloads

  • Running Node.js, Python, PHP, SSR, server actions, or background workers.
  • Building source projects on the server.
  • Databases, writable application storage, secrets, or environment variables.
  • Accounts, dashboards, ownership, updates, custom TTLs, or manual deletion.
  • Directory browsing when a ZIP has no root index.html; reject it with a useful error instead.

The agent's final action is to present url and state the expiration date. UUIDv7 is a good default: high entropy, sortable by creation time, and supported without a central sequence.

Minimal HTTP interface

Four public routes are enough

Keep the application contract small and describe every error in machine-readable JSON.

POST/api/v1/publish

Streams the request body to a temporary area. The Content-Type selects HTML, Markdown, or ZIP. Optional query: spa=1. Returns 201 only after the deployment is fully validated and atomically made live.

GET/

Human- and agent-readable documentation with exact examples, limits, expiration behavior, supported file types, and the static-only boundary.

GET/openapi.json

OpenAPI 3.1 specification. Also expose /llms.txt and /.well-known/agent.json so agents can discover the service without scraping.

GET/healthz

Returns health, writable-volume status, free disk threshold, and cleanup-loop freshness. Do not expose deployment listings or filesystem paths.

Error shape

{
  "error": {
    "code": "invalid_archive",
    "message": "ZIP must contain index.html at its root",
    "details": { "detectedRoot": "my-app/dist/" },
    "docsUrl": "https://publish.tools.company/#zip-layout"
  }
}

Use stable codes such as unsupported_media_type, upload_too_large, too_many_files, invalid_archive, missing_index, rate_limited, insufficient_storage, and service_unavailable.

Single-container architecture

One process, one persistent volume

The company ingress and DNS are existing infrastructure, not additional application containers.

Agent or CI

Builds the front end, packages output, sends one HTTP POST, receives JSON.

→
Company ingress + one Go process

TLS, request limits, upload validation, Markdown rendering, static serving, SPA fallback, health checks, and expiry sweep.

→
Mounted /data volume

sites/<id>/content, a small metadata file, temporary uploads, and cleanup state.

Recommended stack

Go static binary

  • Standard net/http server and archive/zip.
  • goldmark for Markdown.
  • Filesystem metadata per deployment; no database for the MVP.
  • Periodic directory scan is adequate for thousands of temporary sites.
  • Run as a non-root user in a minimal image.
Runtime settings

Container contract

  • PORT=8080
  • DATA_DIR=/data
  • TTL=720h
  • PUBLIC_API_URL=https://publish.tools.company
  • PREVIEW_DOMAIN=preview.tools.company
  • MAX_UPLOAD_BYTES=104857600
  • MAX_EXPANDED_BYTES=524288000

Atomic publish lifecycle

  1. Allocate UUIDv7 and stream the body into /data/tmp/<id>; never buffer an entire archive in memory.
  2. Validate content type, compressed size, expanded size, path safety, file count, and required entry point.
  3. Render Markdown or extract ZIP into a staging content directory.
  4. Write immutable metadata with createdAt, expiresAt, byte count, file count, and SPA mode.
  5. Atomically rename staging to /data/sites/<id>. Only then return 201.
  6. Sweep expired sites hourly and once at startup. Delete content and metadata idempotently; log the site ID, age, and bytes reclaimed.

Security and operational limits

No login does not mean no boundary

The service accepts active HTML and JavaScript. Treat every published site as untrusted, even on a company network.

Use a separate preview origin. Put documentation and publishing at publish.tools.company, and uploaded content at <id>.preview.tools.company. Both can reach the same container. This prevents an uploaded script from sharing cookies, local storage, or service-worker scope with the publishing interface.

Archive safety

Reject dangerous ZIPs

Reject absolute paths, .. traversal, NUL bytes, symlinks, devices, duplicate normalized paths, excessive nesting, more than 10,000 files, and expansion beyond 500 MB.

Resource safety

Bound anonymous writes

Default to 100 MB uploaded, 500 MB expanded, 60 publishes/hour/IP, request timeouts, concurrent-upload limits, and a minimum free-disk threshold.

Network safety

Keep it internal

Allow the publish host only from the corporate network or VPN at the ingress. If it ever becomes internet-facing, add at least a shared bearer token before launch.

Serving headers

  • X-Content-Type-Options: nosniff
  • Referrer-Policy: no-referrer
  • Permissions-Policy disabling camera, microphone, geolocation, and payment by default.
  • Cache-Control: public, max-age=31536000, immutable for versioned assets; shorter caching for HTML.
  • No cookies and no credentialed CORS on the preview domain; no CORS permission on the publish endpoint.
  • Route by host: the publish host never serves uploaded files, and preview hosts never expose /api or documentation routes.

Why a path URL is a compromise

A path such as preview.tools.company/<uuid>/ is operationally easy, but front ends built with root-relative URLs such as /assets/app.js will request the wrong location. Client routers and fetch('/api') have the same problem.

If wildcard DNS/TLS is unavailable, path mode should require builds configured for a relative or explicit base path. HTML rewriting alone cannot reliably fix runtime-generated URLs.

Implementation sequence

Four small phases

A production-ready internal MVP is roughly 3–5 engineering days once DNS, ingress, and volume choices are known.

Lock deployment assumptions · half day

Choose wildcard subdomains versus path mode, the persistent-volume location, maximum bundle size, and the corporate network policy. Confirm that TLS terminates at the existing ingress.

Build the vertical slice · 1–2 days

Implement the Go server, all three request content types, Markdown shell, safe ZIP extraction, atomic publication, static serving, SPA fallback, JSON errors, and the 30-day metadata contract.

Harden and operate · 1–2 days

Add rate and concurrency limits, disk-watermark handling, expiry sweeps, startup recovery, structured logs, metrics, non-root Docker packaging, health checks, and malicious-archive tests.

Make it obvious to agents · half to one day

Ship the documentation landing page, OpenAPI file, discovery documents, exact curl recipes, and an optional 30-line publish.sh helper that ZIPs a directory and prints only the resulting URL.

Later, only if needed

Update/delete capability tokens, selectable TTLs, manual expiry extension, an internal deployment list, object-storage backends, custom domains, password protection, analytics, and per-team quotas. These should not delay the initial static-publish workflow.

Definition of done

Acceptance criteria

Tests should exercise the published result through its real URL, not only the upload response.

  • A raw HTML upload returns a URL and renders exactly that document.
  • A Markdown upload renders headings, tables, lists, code blocks, and links safely.
  • A zipped Vite build loads JavaScript, CSS, fonts, images, and dynamic chunks.
  • SPA mode reloads a nested route and returns index.html.
  • A non-SPA unknown path returns 404.
  • Content survives a container restart when the same volume is mounted.
  • A shortened test TTL removes content and changes its URL to 404 or 410.
  • Path traversal, symlink, ZIP bomb, duplicate path, and over-limit archives are rejected.
  • Partial or disconnected uploads never create a visible deployment.
  • Low-disk mode rejects new uploads while existing sites remain readable.
  • The preview origin has no publishing cookies and cannot call credentialed control APIs.
  • An agent can discover and complete publishing using only the landing page or OpenAPI file.
